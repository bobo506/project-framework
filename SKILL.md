# Project Framework

## 目标

为任意本地项目维护一个可恢复的工作现场：开工时快速知道项目是什么、当前做到哪里、下一步该做什么；收工时把完成内容、验证结果、遗留问题和后续计划写清楚。

本 skill 只规定项目级信息如何组织和流转，不规定具体技术栈、代码风格或业务实现细节。

## 总原则

- 先尊重已有项目规则，再补齐缺失规范。
- 先识别当前 agent 的角色边界，再决定哪些文档可以主动修改。
- `AGENTS.md` 或 `CLAUDE.md` 是项目内 agent 角色与协作边界的首选权威来源；项目开始时必须读取并确认当前角色。Claude Code 读取 `CLAUDE.md`，Codex 和 pi 读取 `AGENTS.md`，pi 同时识别两者。
- 进度文件是最常维护的文件；产品方向、架构决策、模块边界等文档只有在角色允许或用户明确授权时修改。
- 每个开发切片完成后都要汇总进度；记录必须基于本次真实对话、实际文件变化和实际验证结果，不要补写没有发生的事项。
- `PROGRESS.md` 保持为短小当前快照；已完成切片的完整记录归入 `PROGRESS_HISTORY.md` 或项目既有历史进度文件。
- 开工恢复只汇报状态和建议下一步；除非用户明确要求，不要自动启动服务、杀进程或修改代码。

## 项目宪法（须写入项目必读文档）

以下规则是项目级"宪法"——无论哪个 agent、无论是否加载本 skill，都必须遵守。在项目初始化时**必须**将这些规则写入项目必读文档（`AGENTS.md` 或 `CLAUDE.md`）。

**写入目标判断规则：**
- Claude Code 项目写入 `CLAUDE.md`
- Codex / pi 项目写入 `AGENTS.md`
- pi 同时识别 `AGENTS.md` 和 `CLAUDE.md`
- 若无法判断当前项目主要使用哪个 agent，则询问用户后再决定

**必须写入的规则清单：**

### 1) Agent 角色体系

- **决策 Agent**：在方向明确、无重大不确定项时，可自主连续开发多个切片；遇到产品方向、破坏性接口/数据变更、部署/权限风险或用户体验重大分歧时再确认。可维护项目简报、进度、决策、路线图、上下文和模块规划等上层文档。
- **执行 Agent**：每开发一个切片后必须暂停，汇总进度，并等待决策 Agent 或项目管理者确认下一切片。主要维护代码、测试和被授权的局部文档；默认可更新进度文件。
- **审查/验证 Agent**：优先输出风险、验证结果和建议；默认不改方向性文档。
- 如果项目已有角色规则，以项目本地规则为准。如果角色不明确，修改方向性文档前先确认。

### 2) 切片刹车规则

- 决策 Agent 每次恢复默认最多连续推进 2 个明确切片。
- 若都是小型文档/配置切片，可最多 3 个。
- 若涉及前端真实交互、数据库/协议、部署/权限或跨模块协作，默认 1 个切片后暂停汇总。
- 决策 Agent 连续工作约 60-90 分钟后，不应开启新切片；应先完成当前切片的必要验证、汇总进度、提交/推送，并输出下一步建议。
- 若后续任务需要重新读取另一个模块文档，或 Agent 明显开始依赖"回忆前文"才能继续判断，应在完成当前切片收尾后暂停，不再开启新切片。
- 执行 Agent 每开发一个切片后必须暂停，汇总进度，等待确认。
- 任意角色：每完成一个可能影响功能的切片，都要运行必要验证、汇总进度，并按项目规则决定 git/GitHub 提交。

### 3) 上下文交接原则

- 核心约束：**先汇总、先落盘、先提交，再清上下文**；不可反过来。
- 当上下文达到 80%-90% 或明显接近承载上限时，先执行汇总进度，确认进度文件已写入，完成必要验证与提交，再清除上下文。
- 若运行环境提供 5 小时额度或 token 用量占比，且当前占比达到或超过 90%，不要开启新切片；先完成当前切片收尾、汇总进度、提交/推送，并输出 `继续项目`。
- 如果环境没有暴露精确占比，改用批次上限、60-90 分钟工作时长、上下文接近上限、模块文档切换和"需要回忆前文"这些可观察信号来节制连续开发。

### 4) 人工验收门禁

- 当完成一个可独立体验的里程碑版本时（例如登录模块完整可操作、一个 Agent 可接入并完成消息接收/反馈、Group Hall 可完整创建/进入/发消息），暂停继续扩展新功能。
- 执行汇总进度，完成必要验证、git commit，并在可用时推送 GitHub 或创建 PR。
- 用中文提供验收说明、启动方式、测试账号/入口、已知限制。
- 等待项目管理者人工验收后再继续下一阶段。

### 5) 进度文件维护规则

- `PROGRESS.md` 保持为短小当前快照，每次更新 `Latest` 小节；不要长期累积完整历史。
- 已完成切片的完整记录追加到 `PROGRESS_HISTORY.md` 顶部（或项目既有历史进度文件）。
- 记录必须基于本次真实对话、实际文件变化和实际验证结果，不要补写没有发生的事项。

### 6) GitHub 提交规范

- GitHub 的人类可读描述使用中文，包括 PR 标题、PR 描述、变更摘要、测试说明和验收说明。
- 决策 Agent 可自主判断是否继续下一切片，但每个切片完成后应提交一个可回溯版本。
- 执行 Agent 开发的切片应先提交给决策 Agent 或项目管理者判断是否提交。

### 7) 当前 Agent 角色记录

`AGENTS.md` 或 `CLAUDE.md` 中必须保留以下固定小节：

```markdown
## 当前 Agent 角色

- 当前 Codex 角色：[决策 Agent / 执行 Agent / 审查 Agent]
- 当前 Claude 角色：[决策 Agent / 执行 Agent / 审查 Agent]
- 当前 pi 角色：[决策 Agent / 执行 Agent / 审查 Agent]
- 如项目管理者人工修改本节，后续 Agent 必须按最新角色执行。
```

## 项目识别

按顺序确定项目根目录：

1. 用户明确指定的路径。
2. 当前工作目录所属的 Git 根目录。
3. 含有项目标记的最近祖先目录：`CLAUDE.md`、`AGENTS.md`、`README.md`、`package.json`、`pyproject.toml`、`requirements.txt`、`Cargo.toml`、`go.mod`、`docs/PROGRESS.md`、`docs/进度.md`。
4. 如果当前目录像多个项目的父目录，询问用户选择具体项目。
5. 若仍不确定，使用当前工作目录并在回复中说明不确定性。

## 本地规则与角色

进入项目后优先读取轻量入口文档：

- `AGENTS.md`
- `CLAUDE.md`
- `README.md`
- `docs/PROJECT_BRIEF.md`
- `docs/项目简报.md`

判断当前工作身份：

- 项目管理/决策角色：可维护项目简报、进度、决策、路线图、上下文和模块规划等上层文档。
- 施工/实现角色：主要维护代码、测试和被授权的局部文档；默认可更新进度文件。
- 审查/验证角色：优先输出风险、验证结果和建议；默认不改方向性文档。

如果项目已有角色规则（参见 [项目宪法](#项目宪法须写入项目必读文档) 中写入的规则），以项目本地规则为准。如果角色不明确，修改方向性文档前先确认。

详细角色节奏参见项目宪法中的"切片刹车规则"。

## 推荐文档结构

新项目或缺少文档规范的项目，推荐使用以下轻量结构；已有命名体系时优先沿用已有文件。

```text
docs/
  PROJECT_BRIEF.md      # 项目公共上下文：定位、架构、技术栈、目录、模块索引
  PROGRESS.md           # 当前快照、卡点、下一步、最近恢复所需信息
  PROGRESS_HISTORY.md   # 已完成切片的完整历史记录
  DECISIONS.md          # 关键决策与理由，可选
  SETUP.md              # 环境、依赖、启动、端口、验证命令，可选
  MODULE_*.md           # 模块级说明，可选
```

中文项目可使用对应中文名，例如 `项目简报.md`、`进度.md`、`决策记录.md`、`启动说明.md`。不要为了统一命名而重命名已有项目文档，除非用户明确要求。

## 进度文件发现

按顺序寻找进度文件：

1. 用户明确指定的路径。
2. 常见路径：`docs/PROGRESS.md`、`docs/进度.md`、`PROGRESS.md`、`进度.md`、`docs/status.md`、`STATUS.md`。
3. Markdown 内容特征匹配：包含 `Current Progress`、`Open Questions`、`Next Plan`、`当前阶段`、`已完成`、`待确认`、`下一步`、`最后更新`。
4. 如果找不到，建议创建 `docs/PROGRESS.md`；中文项目可建议 `docs/进度.md`。

如果存在多个候选且无法判断，列出候选并询问用户选择。

## 标准进度格式

优先保持项目已有格式。新建或重构时使用：

```markdown
# Project Progress

## Latest
Updated: YYYY-MM-DD HH:mm (Timezone)

### 1) Current Progress
- ...

### 2) Open Questions / Pending Confirmation
- ...

### 3) Next Plan
1. ...

### 4) Verification
- Run: `...`
- Result: ...
- Not verified: ...

### 5) Changed Files
- `path/to/file`

## Recent Notes
- 完整历史见 `docs/PROGRESS_HISTORY.md`。
```

中文项目可以将标题本地化，但保持信息等价。

历史进度文件可使用：

```markdown
# Project Progress History

## YYYY-MM-DD HH:mm (Timezone)
### Current Progress
- ...
### Open Questions / Pending Confirmation
- ...
### Next Plan
1. ...
### Verification
- ...
### Changed Files
- ...
```

## 开工恢复流程

触发词示例：继续项目、恢复进度、读取进度、从上次继续、项目进展到哪了、打开这个项目。

执行：

1. 识别项目根目录，并说明依据。
2. 读取本地规则文件（`AGENTS.md` 或 `CLAUDE.md`）以确认角色边界。
3. 找到并读取进度文件。
4. 必要时读取项目简报或 README 来补全上下文。
5. 在进度快照中确认或记录本轮启动角色；角色来源以 `AGENTS.md` 或 `CLAUDE.md` 的"当前 Agent 角色"小节为准。
6. 用不超过 10 行汇报：
   - 当前项目
   - 进度文件
   - 当前 agent 角色
   - 当前阶段/当前进展
   - 上次完成的关键事项，不超过 3 条
   - 当前卡点或待确认问题
   - 建议下一步
7. 询问用户是否按建议继续，或是否有新的优先级。

不要在用户确认前自动启动服务、清理端口、安装依赖或改代码。

## 汇总进度与切片收尾流程

触发词示例：汇总进度、收工、记录进度、保存项目状态、总结今天、汇总今日情况、更新进度文件、写入项目进度。每个开发切片完成后也必须执行本流程。

执行：

1. 回顾本次会话和实际操作，梳理：
   - 完成事项
   - 修改文件
   - 验证命令与结果
   - 未验证内容
   - 新增卡点和待确认问题
   - 下一步计划
2. 读取当前进度文件。
3. 更新 `PROGRESS.md` 的 `Latest` 为短小当前快照，并记录本轮启动角色。
4. 将本次已完成切片的完整记录追加到 `PROGRESS_HISTORY.md` 顶部；不要让 `PROGRESS.md` 长期累积完整历史。
5. 如本轮有代码/文档改动，运行必要验证；验证结果写入进度。
6. 按项目角色规则处理提交（参见项目宪法中的"GitHub 提交规范"和"切片刹车规则"）。
7. 写入后简短确认：进度已更新，并说明下次可说"继续项目"恢复。

如果本次没有做实际项目工作，不要伪造进度；可以只记录讨论出的规则、决策或待办。

## 上下文交接/清除流程

当上下文达到 80%-90% 或明显接近承载上限时：

1. 先执行"汇总进度"。
2. 确认进度文件已写入；如有代码/文档改动，完成必要验证与提交。
3. 如果环境支持自动清除上下文或开启新窗口，则自动执行。
4. 如果不支持，则输出明确恢复指令：`继续项目`。

核心约束与详细规则参见项目宪法中的"上下文交接原则"。

## 人工验收门禁

当完成一个可独立体验的里程碑版本时：

1. 暂停继续扩展新功能。
2. 执行"汇总进度"。
3. 完成必要验证、git commit，并在可用时推送 GitHub 或创建 PR。
4. 用中文提供验收说明、启动方式、测试账号/入口、已知限制。
5. 等待项目管理者人工验收后再继续下一阶段。

详细规则参见项目宪法中的"人工验收门禁"。

## 初始化项目文档流程

触发词示例：初始化项目文档、建立项目规范、创建项目框架、给这个项目加进度管理。

执行：

1. 识别项目根目录。
2. 检查已有 `docs/`、`README.md`、`CLAUDE.md`、`AGENTS.md`。
3. 如果已有文档体系，先汇报现状并建议最小补齐项。
4. 如果缺少项目必读文档，按写入目标判断规则创建：
   - Claude Code 项目创建或更新 `CLAUDE.md`
   - Codex / pi 项目创建或更新 `AGENTS.md`
   - 无法判断时询问用户
5. **创建或更新项目必读文档时，必须将"项目宪法"中的全部 7 条规则写入文件。** 具体模板参见下方[附录](#附录agentsmd--claudemd-项目宪法模板)。
6. 创建轻量文档结构：
   - `docs/PROJECT_BRIEF.md`
   - `docs/PROGRESS.md`
   - `docs/PROGRESS_HISTORY.md`
   - 可选 `docs/DECISIONS.md`
   - 可选 `docs/SETUP.md`
7. 若目标文件已存在，先读取原内容，检查是否缺少宪法条款。若缺少，向用户汇报缺失项并建议补齐；补齐时以增量方式写入，保留原有项目特有规则。
8. 不要覆盖已有文件中的项目特有内容；只补齐缺失的宪法条款。

## 决策与文档同步

当用户明确做出产品、架构、流程或角色决策时：

- 决策角色可以同步到 `DECISIONS.md`、`PROJECT_BRIEF.md` 或相关模块文档。
- 施工角色只在授权时同步方向性文档；否则写入进度文件的待确认/决策摘要。
- 决策记录应包含：日期、决策、理由、影响范围、后续动作。

## TALK 原型经验

TALK 项目的经验可作为默认参考：

- 使用 `1+N` 文档结构：项目入口规则 + 公共简报 + 模块文档 + 进度文档。
- `PROJECT_BRIEF.md` 存公共上下文和模块索引。
- `MODULE_*.md` 存模块范围、接口契约、现状和验收标准。
- `PROGRESS.md` 存当前快照和最近历史。
- `PROGRESS_HISTORY.md` 存较旧历史和决策背景。
- `AGENTS.md` / `CLAUDE.md` 可表达不同 agent 的角色边界。

这些是推荐模式，不是强制模板。

- 决策 Agent 连续工作约 60-90 分钟后，不应开启新切片；应先完成当前切片的必要验证、汇总进度、提交/推送，并输出下一步建议
- 若后续任务需要重新读取另一个模块文档，或 Agent 明显开始依赖"回忆前文"才能继续判断，应在完成当前切片收尾后暂停，不再开启新切片
- 执行 Agent 每开发一个切片后必须暂停，汇总进度，等待确认
- 任意角色：每完成一个可能影响功能的切片，都要运行必要验证、汇总进度，并按项目规则决定 git/GitHub 提交

## 3) 上下文交接原则

- 核心约束：先汇总、先落盘、先提交，再清上下文；不可反过来
- 当上下文达到 80%-90% 或明显接近承载上限时，先执行汇总进度，确认进度文件已写入，完成必要验证与提交，再清除上下文
- 若运行环境提供 token 用量占比，且当前占比达到或超过 90%，不要开启新切片；先完成当前切片收尾、汇总进度、提交/推送，并输出"继续项目"以恢复
- 如果环境没有暴露精确占比，改用批次上限、60-90 分钟工作时长、上下文接近上限、模块文档切换和"需要回忆前文"这些可观察信号来节制连续开发

## 4) 人工验收门禁

- 当完成一个可独立体验的里程碑版本时，暂停继续扩展新功能
- 执行汇总进度，完成必要验证、git commit，并在可用时推送 GitHub 或创建 PR
- 用中文提供验收说明、启动方式、测试账号/入口、已知限制
- 等待项目管理者人工验收后再继续下一阶段

## 5) 进度文件维护规则

- `PROGRESS.md` 保持为短小当前快照，每次更新 `Latest` 小节
- 已完成切片的完整记录追加到 `PROGRESS_HISTORY.md` 顶部（或项目既有历史进度文件）
- 记录必须基于本次真实对话、实际文件变化和实际验证结果，不要补写没有发生的事项

## 6) GitHub 提交规范

- GitHub 的人类可读描述使用中文：PR 标题、PR 描述、变更摘要、测试说明、验收说明
- 决策 Agent 可自主判断是否继续下一切片，但每个切片完成后应提交一个可回溯版本
- 执行 Agent 开发的切片应先提交给决策 Agent 或项目管理者判断是否提交

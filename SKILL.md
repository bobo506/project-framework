# Project Framework

## 目标

为任意本地项目维护一个可恢复的工作现场：开工时快速知道项目是什么、当前做到哪里、下一步该做什么；收工时把完成内容、验证结果、遗留问题和后续计划写清楚。

本 skill 只规定项目级信息如何组织和流转，不规定具体技术栈、代码风格或业务实现细节。

## 总原则

- 先尊重已有项目规则，再补齐缺失规范。
- 先识别当前 agent 的角色边界，再决定哪些文档可以主动修改。
- 进度文件是最常维护的文件；产品方向、架构决策、模块边界等文档只有在角色允许或用户明确授权时修改。
- 收工记录必须基于本次真实对话、实际文件变化和实际验证结果，不要补写没有发生的事项。
- 开工恢复只汇报状态和建议下一步；除非用户明确要求，不要自动启动服务、杀进程或修改代码。

## 项目识别

按顺序确定项目根目录：

1. 用户明确指定的路径。
2. 当前工作目录所属的 Git 根目录。
3. 含有项目标记的最近祖先目录：`CLAUDE.md`、`AGENTS.md`、`README.md`、`package.json`、`pyproject.toml`、`requirements.txt`、`Cargo.toml`、`go.mod`、`docs/PROGRESS.md`、`docs/进度.md`。
4. 如果当前目录像多个项目的父目录，询问用户选择具体项目。
5. 若仍不确定，使用当前工作目录并在回复中说明不确定性。

## 本地规则与角色

进入项目后优先读取轻量入口文档：

- `AGENTS.md`
- `CLAUDE.md`
- `README.md`
- `docs/PROJECT_BRIEF.md`
- `docs/项目简报.md`

判断当前工作身份：

- 项目管理/决策角色：可维护项目简报、进度、决策、路线图、上下文和模块规划等上层文档。
- 施工/实现角色：主要维护代码、测试和被授权的局部文档；默认可更新进度文件。
- 审查/验证角色：优先输出风险、验证结果和建议；默认不改方向性文档。

如果项目已有角色规则，以项目本地规则为准。如果角色不明确，修改方向性文档前先确认。

## 推荐文档结构

新项目或缺少文档规范的项目，推荐使用以下轻量结构；已有命名体系时优先沿用已有文件。

```text
docs/
  PROJECT_BRIEF.md      # 项目公共上下文：定位、架构、技术栈、目录、模块索引
  PROGRESS.md           # 当前进度、卡点、下一步、最近历史
  PROGRESS_HISTORY.md   # 较旧历史归档，可选
  DECISIONS.md          # 关键决策与理由，可选
  SETUP.md              # 环境、依赖、启动、端口、验证命令，可选
  MODULE_*.md           # 模块级说明，可选
```

中文项目可使用对应中文名，例如 `项目简报.md`、`进度.md`、`决策记录.md`、`启动说明.md`。不要为了统一命名而重命名已有项目文档，除非用户明确要求。

## 进度文件发现

按顺序寻找进度文件：

1. 用户明确指定的路径。
2. 常见路径：`docs/PROGRESS.md`、`docs/进度.md`、`PROGRESS.md`、`进度.md`、`docs/status.md`、`STATUS.md`。
3. Markdown 内容特征匹配：包含 `Current Progress`、`Open Questions`、`Next Plan`、`当前阶段`、`已完成`、`待确认`、`下一步`、`最后更新`。
4. 如果找不到，建议创建 `docs/PROGRESS.md`；中文项目可建议 `docs/进度.md`。

如果存在多个候选且无法判断，列出候选并询问用户选择。

## 标准进度格式

优先保持项目已有格式。新建或重构时使用：

```markdown
# Project Progress

## Latest
Updated: YYYY-MM-DD HH:mm (Timezone)

### 1) Current Progress
- ...

### 2) Open Questions / Pending Confirmation
- ...

### 3) Next Plan
1. ...

### 4) Verification
- Run: `...`
- Result: ...
- Not verified: ...

### 5) Changed Files
- `path/to/file`

## History

### YYYY-MM-DD HH:mm (Timezone)
#### Current Progress
- ...
#### Open Questions / Pending Confirmation
- ...
#### Next Plan
1. ...
#### Verification
- ...
#### Changed Files
- ...
```

中文项目可以将标题本地化，但保持信息等价。

## 开工恢复流程

触发词示例：继续项目、恢复进度、读取进度、从上次继续、项目进展到哪了、打开这个项目。

执行：

1. 识别项目根目录，并说明依据。
2. 读取本地规则与角色边界。
3. 找到并读取进度文件。
4. 必要时读取项目简报或 README 来补全上下文。
5. 用不超过 10 行汇报：
   - 当前项目
   - 进度文件
   - 当前阶段/当前进展
   - 上次完成的关键事项，不超过 3 条
   - 当前卡点或待确认问题
   - 建议下一步
6. 询问用户是否按建议继续，或是否有新的优先级。

不要在用户确认前自动启动服务、清理端口、安装依赖或改代码。

## 收工记录流程

触发词示例：收工、记录进度、保存项目状态、总结今天、汇总今日情况、更新进度文件、写入项目进度。

执行：

1. 回顾本次会话和实际操作，梳理：
   - 完成事项
   - 修改文件
   - 验证命令与结果
   - 未验证内容
   - 新增卡点和待确认问题
   - 下一步计划
2. 读取当前进度文件。
3. 更新 `Latest` 为当前快照。
4. 在 `History` 顶部追加本次记录；不要删除历史。
5. 如历史过长，可建议归档到 `PROGRESS_HISTORY.md`，但不要擅自大规模移动内容。
6. 写入后简短确认：进度已更新，并说明下次可说“继续项目”恢复。

如果本次没有做实际项目工作，不要伪造进度；可以只记录讨论出的规则、决策或待办。

## 初始化项目文档流程

触发词示例：初始化项目文档、建立项目规范、创建项目框架、给这个项目加进度管理。

执行：

1. 识别项目根目录。
2. 检查已有 `docs/`、`README.md`、`CLAUDE.md`、`AGENTS.md`。
3. 如果已有文档体系，先汇报现状并建议最小补齐项。
4. 如果缺少文档，创建轻量模板：
   - `docs/PROJECT_BRIEF.md`
   - `docs/PROGRESS.md`
   - 可选 `docs/DECISIONS.md`
   - 可选 `docs/SETUP.md`
5. 不要覆盖已有文件。若目标文件已存在，先读取并增量补齐。

## 决策与文档同步

当用户明确做出产品、架构、流程或角色决策时：

- 决策角色可以同步到 `DECISIONS.md`、`PROJECT_BRIEF.md` 或相关模块文档。
- 施工角色只在授权时同步方向性文档；否则写入进度文件的待确认/决策摘要。
- 决策记录应包含：日期、决策、理由、影响范围、后续动作。

## TALK 原型经验

TALK 项目的经验可作为默认参考：

- 使用 `1+N` 文档结构：项目入口规则 + 公共简报 + 模块文档 + 进度文档。
- `PROJECT_BRIEF.md` 存公共上下文和模块索引。
- `MODULE_*.md` 存模块范围、接口契约、现状和验收标准。
- `PROGRESS.md` 存当前快照和最近历史。
- `PROGRESS_HISTORY.md` 存较旧历史和决策背景。
- `AGENTS.md` / `CLAUDE.md` 可表达不同 agent 的角色边界。

这些是推荐模式，不是强制模板。
